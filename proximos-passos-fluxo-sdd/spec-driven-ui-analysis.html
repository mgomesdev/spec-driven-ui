01 — visão geral

O que é o projeto

O spec-driven-ui é um sistema de geração de UI frontend orientado a specs, estruturado em quatro etapas sequenciais com aprovação humana entre cada fase. O fluxo completo: research → plan → tasks → orchestrator — cada etapa produzindo um artefato Markdown que alimenta a próxima.

A ideia central é sólida e alinha-se com as melhores práticas emergentes do ecossistema de SDD (Spec-Driven Development). O repositório usa Claude Code (com skills em .agent/skills/) como agente executor, Next.js/React como stack alvo, e o progress.md como memória acumulativa entre iterações.

02 — diagnóstico

Problemas identificados

Crítico
Sem orçamento de tokens explícito por task
O tasks.md não especifica o tamanho estimado de contexto de cada história. O orchestrator pode selecionar uma história que excede a janela disponível, causando truncamento silencioso.
Crítico
Orchestrator lê plan.md inteiro a cada iteração
O fluxo instrui o orchestrator a consultar a seção do plan.md referenciada pela task, mas na prática o agente frequentemente puxa o arquivo inteiro, consumindo tokens desnecessariamente.
Crítico
progress.md pode crescer ilimitadamente
Padrões e aprendizados são acumulados indefinidamente. Sem destilação periódica, o arquivo cresce e ocupa contexto valioso, eventualmente degradando qualidade ao ser lido pelo agente.
Alto
Ausência de constitution.md / invariantes do projeto
Não há um documento de "constituição" que codifique princípios não-negociáveis (stack, padrões de código, convenções de nomes). O agente pode inventar padrões conflitantes entre features.
Alto
Research sem perguntas de esclarecimento estruturadas
A skill de research "faz perguntas se necessário", mas não tem protocolo para identificar lacunas críticas. Informações faltantes (auth, i18n, responsividade) só emergem tarde no plano ou implementação.
Alto
Sem verificação de consistência cross-artefato
Não há etapa que valide se os tipos do plan.md batem com os contratos do research.md, ou se as tasks cobrem todos os critérios de aceitação do research.
Médio
Tasks sem critério de "janela única"
O README menciona que cada história deve caber em "uma janela de contexto", mas não há definição do que isso significa em termos concretos (linhas, componentes, arquivos).
Médio
Sem estado de rollback das tasks
Se o orchestrator falha no meio de uma task, não há instrução clara para reverter estado parcial. O progress.md pode ficar inconsistente com o código.
Médio
Commit message não normalizado
O padrão feat: US-XXX - Título é bom, mas falta incluir o nome da feature (feat(nome-feature): US-XXX) para rastreabilidade por escopo no git log.
Informacional
Sem suporte a mocks tipados automáticos
O README menciona mocks quando o backend não está pronto, mas as skills não têm instruções para gerar mocks tipados alinhados ao plan.md automaticamente.
Informacional
FLUXO-DEV.md e README duplicam informação
Dois arquivos descrevem o mesmo fluxo com pequenas variações. Risco de divergência e confusão para o agente sobre qual é a fonte de verdade.
Informacional
Sem indicadores de qualidade nos checkpoints
O orchestrator roda typecheck, mas não há critério mínimo de cobertura, nem verificação de acessibilidade ou lighthouse score para UIs.
03 — janela de contexto

Gestão de contexto & tokens

Problema central: o orchestrator opera em loop, e a cada iteração carrega o histórico de mensagens + progress.md + tasks.md + seção do plan.md. Sem controles explícitos, o contexto cresce de forma quadrática — especialmente em features grandes.

Melhoria 1 — Token Budget por Task

Adicionar ao template de tasks.md um campo ctx_estimate em cada história, calculado na geração pelo agente de tasks:

## US-003 — Formulário de criação
Passes: false
Priority: 3
ctx_estimate: ~2k tokens   # novo campo
plan_ref: "#section-form-creation"
files_affected:
  - src/app/components/CreateForm.tsx
  - src/app/hooks/useCreateForm.ts
AC:
  - [ ] Validação client-side com zod
  - [ ] Estado de loading/erro visível
tasks.md

O orchestrator então verifica: contexto atual + ctx_estimate ≤ 60% da janela disponível antes de iniciar. Se exceder, ele compacta o histórico ou sugere nova sessão.

Melhoria 2 — Leitura Cirúrgica do plan.md

Em vez de carregar o plan.md inteiro, o orchestrator deve ser instruído a:

# Na skill do orchestrator:
REGRA: Ao iniciar uma task, leia APENAS:
  1. A seção exata referenciada em plan_ref
  2. A seção de "Tipos globais" (sempre inclusa)
  3. O arquivo progress.md (máx 100 linhas)

NUNCA carregue o plan.md inteiro.
Se precisar de outra seção, pare e declare qual.
SKILL.md orchestrator

Melhoria 3 — Destilação periódica do progress.md

Adicionar ao fluxo do orchestrator uma verificação: se progress.md > 150 linhas, antes de prosseguir execute a sub-rotina de destilação.

## Sub-rotina: destilar progress.md
QUANDO: progress.md > 150 linhas
AÇÃO:
  1. Identifique padrões repetidos ou redundantes
  2. Consolide em seção "Padrões confirmados" (máx 30 linhas)
  3. Mova detalhes históricos para progress-archive.md
  4. Mantenha progress.md abaixo de 80 linhas operacionais
orchestrator flow
04 — prevenção de alucinações

Reduzindo alucinações

Alucinações em sistemas agênticos ocorrem principalmente em três cenários: contexto insuficiente (o modelo inventa o que não sabe), ambiguidade nos artefatos (interpretação livre de especificações vagas), e loop sem grounding (o modelo confirma a si mesmo).

Melhoria 4 — constitution.md (Invariantes do Projeto)

Criar um arquivo .agent/constitution.md que é SEMPRE injetado no início de cada skill. Este arquivo codifica o que nunca muda:

# PROJECT CONSTITUTION — lido em TODA invocação de skill

Stack obrigatório:
  - Next.js 15 (App Router) — NÃO Pages Router
  - TypeScript strict mode
  - Tailwind CSS — NÃO CSS Modules, NÃO styled-components
  - Zod para validação de formulários
  - React Query para server state

Proibições absolutas:
  - ❌ NÃO use `any` em TypeScript
  - ❌ NÃO crie componentes com lógica de negócio e UI juntas
  - ❌ NÃO chame APIs diretamente em componentes — use hooks/services
  - ❌ NÃO use `useState` para server state — use React Query

Convenções de nomes:
  - Componentes: PascalCase em src/app/components/
  - Hooks: camelCase prefixado com "use" em src/app/hooks/
  - Types: PascalCase com sufixo "Type" ou "Props"
  - Services: camelCase em src/app/services/

Critérios de aceite mínimos:
  - Typecheck passando (tsc --noEmit)
  - Zero console.error no browser
  - Responsivo: mobile (320px) e desktop (1280px)
.agent/constitution.md

Melhoria 5 — Perguntas de Esclarecimento Estruturadas no Research

A skill de research deve ter um checklist de domínios críticos para explorar antes de gerar o artefato:

## Protocolo de esclarecimento — skill research

ANTES de gerar research.md, confirme:

Autenticação:
  - Esta feature requer usuário autenticado? Qual provider?

Dados:
  - Quais endpoints de API serão consumidos?
  - Formato de paginação/filtro?

Estado:
  - Dados são persistidos localmente ou só server-side?
  - Há otimistic updates?

Responsividade:
  - Breakpoints específicos além do padrão Tailwind?

Acessibilidade:
  - Há requisitos WCAG? Nível A, AA?

i18n:
  - Feature precisa de internacionalização?

Se qualquer item acima for ambíguo, PARE e pergunte.
NÃO assuma. NÃO invente defaults silenciosos.
skill research — protocolo

Melhoria 6 — Validação Cruzada de Artefatos (Cross-check)

Adicionar uma nova micro-etapa opcional mas recomendada entre tasks e orchestrator: uma verificação de consistência.

Fase 1
research.md
US + ACs
Fase 2
plan.md
Tipos + Contratos
Fase 3
tasks.md
Histórias atômicas
Fase 3.5 — novo
cross-check
Consistência AC↔tasks
Fase 4
orchestrator
Implementação
## Skill: cross-check (nova)

INPUT: research.md + plan.md + tasks.md
VERIFICAR:

1. Todo AC do research tem ao menos 1 task correspondente?
   - Liste ACs sem cobertura: BLOQUEANTE

2. Todo tipo definido no plan é usado em ao menos 1 task?
   - Tipos órfãos: aviso não-bloqueante

3. Os contratos de API do plan batem com os endpoints do research?
   - Divergências: BLOQUEANTE

4. Há tasks que dependem de outras ainda não marcadas como Passes:true?
   - Gere grafo de dependências se detectado

OUTPUT: cross-check.md com lista de issues
HUMANO APROVA: Sim — só avança se zero bloqueantes
skill cross-check

Melhoria 7 — Anti-Hallucination Guards no Orchestrator

Adicionar ao SKILL.md do orchestrator um conjunto de "guards" explícitos:

## Anti-Hallucination Guards — orchestrator

NUNCA:
  - Invente nomes de componentes não listados no plan.md
  - Assuma que uma biblioteca está instalada sem verificar package.json
  - Crie arquivos fora dos paths definidos no plan.md
  - Implemente lógica de negócio não descrita na task atual
  - Marque Passes:true sem ter rodado tsc --noEmit com sucesso

SEMPRE:
  - Se um tipo necessário não existe no plan.md, PARE e reporte
  - Se um endpoint de API não está no plan.md, PARE e reporte
  - Confirme paths de import contra tsconfig.json antes de usar
  - Leia o package.json antes de usar qualquer biblioteca externa

REGRA DE OURO: Em caso de dúvida, reporte ao humano.
NÃO invente. NÃO adivinhe. NÃO assuma.

05 — fluxo revisado

Fluxo completo melhorado

O fluxo proposto adiciona três elementos ao pipeline original: constitution.md como base sempre presente, cross-check como etapa de validação, e compactação de contexto como sub-rotina do orchestrator.

## Fluxo Revisado — Research → Plan → Tasks → Cross-check → Orchestrator

[constitution.md]  ← injetado em TODAS as skills como contexto base
       │
[US do PO]
       │
       ▼
research  →  research.md   →  [👤 aprovação]
       │
       │  ↳ protocolo de esclarecimento estruturado antes de gerar
       ▼
plan      →  plan.md        →  [👤 aprovação]
       │
       │  ↳ seções com âncoras (#section-id) para leitura cirúrgica
       │  ↳ ctx_size estimada por seção
       ▼
tasks     →  tasks.md       →  [👤 aprovação]
       │
       │  ↳ ctx_estimate por task
       │  ↳ files_affected listados
       ▼
cross-check  →  cross-check.md  →  [👤 aprovação — bloqueante se issues]
       │
       │  ↳ AC coverage, tipos órfãos, contratos
       ▼
orchestrator:
  LOOP:
    1. Verifica ctx disponível vs ctx_estimate da próxima task
    2. Se progress.md > 150 linhas → destila
    3. Carrega APENAS seção do plan.md referenciada
    4. Aplica Anti-Hallucination Guards
    5. Implementa → tsc --noEmit → browser check
    6. Commita: feat(nome-feature): US-XXX - Título
    7. Atualiza tasks.md (Passes: true) e progress.md
    8. Verifica próxima task
  FIM LOOP → destila conhecimento → PR

Melhoria 10 — Skill de rollback

Adicionar instrução explícita no orchestrator para falha parcial:

## Tratamento de falha — orchestrator

SE tsc --noEmit falha após implementação:
  1. NÃO marque Passes:true
  2. Registre o erro exato no progress.md
  3. Tente corrigir (máx 2 tentativas)
  4. Se ainda falha: git stash + reporte ao humano
     com: lista de erros + arquivos afetados

SE implementação parcialmente completa ao fim da sessão:
  1. Marque a task como Passes: partial
  2. Liste o que foi feito e o que falta no progress.md
  3. NÃO commite código com erros de tipo

07 — melhorias gerais

Melhorias gerais no projeto

G1
Unificar FLUXO-DEV.md e README.md
Documentação

Atualmente dois arquivos descrevem o mesmo fluxo. O README deve ser a documentação para humanos (visão geral, como contribuir), enquanto FLUXO-DEV.md deve ser a fonte de verdade técnica consumida pelos agentes. Adicionar no topo do README: "Para instruções do agente, veja FLUXO-DEV.md".

G2
Adicionar .agent/constitution.md ao .gitignore team-specific
Estrutura

A constitution.md deve ter duas variantes: uma base commitada no repo (padrões do projeto) e uma local não-commitada para preferências do dev individual. Adicionar .agent/constitution.local.md ao .gitignore.

G3
Adicionar script de scaffold para nova feature
DX

Criar um script scripts/new-feature.sh [nome] que cria automaticamente a estrutura specs/features/[nome]/ com os templates vazios de research, plan, tasks e progress. Elimina erros de estrutura e acelera o início.

#!/bin/bash
FEATURE=$1
DIR="specs/features/$FEATURE"
mkdir -p $DIR
cp .agent/templates/research.template.md $DIR/research.md
cp .agent/templates/plan.template.md     $DIR/plan.md
cp .agent/templates/tasks.template.md    $DIR/tasks.md
echo "# Progress: $FEATURE\n\n## Padrões confirmados\n" > $DIR/progress.md
echo "✅ Feature scaffold criado em $DIR"
scripts/new-feature.sh
G4
Padronizar commit com escopo de feature
Git

Mudar o padrão de commit de feat: US-XXX - Título para feat(nome-feature): US-XXX - Título. Isso permite usar git log --grep="feat(checkout)" para rastrear todos os commits de uma feature específica, além de ser compatível com Conventional Commits para geração automática de CHANGELOG.

G5
Adicionar quality gates ao orchestrator
Qualidade

Além do tsc --noEmit, o orchestrator deve ter um conjunto mínimo de quality gates configurável:

## Quality Gates — orchestrator
# Configurável em .agent/quality-gates.json
{
  "typecheck": true,      // sempre obrigatório
  "lint": true,          // eslint --max-warnings 0
  "tests": false,         // opcional por projeto
  "browser_check": true,  // visual para tasks com UI
  "a11y_basic": false      // axe-core quando habilitado
}
.agent/quality-gates.json
G6
Índice de features em specs/INDEX.md
Navegação

À medida que o projeto cresce, o diretório specs/features/ acumula features. Adicionar um specs/INDEX.md mantido automaticamente pelo orchestrator ao fim de cada feature, listando: nome, status (em andamento / concluída), data, US cobertos.

G7
Suporte a mocks tipados automáticos no plan
Backend-parity

Quando contratos de API estão marcados como "a confirmar" no research.md, a skill de plan deve gerar automaticamente uma seção ## Mocks com factories tipadas usando os tipos definidos. O orchestrator implementa com os mocks; quando o backend entregar, apenas os mocks são substituídos — zero mudanças de tipo.
